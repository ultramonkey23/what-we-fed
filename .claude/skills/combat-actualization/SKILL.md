# Combat Actualization

Use before implementation when a combat request is aspirational, broad, or design-heavy.

## Procedure
1. Translate the request into one player-visible combat payoff.
2. Verify what already exists in repo truth before naming gaps.
3. Identify the runtime owner and the smallest viable patch target.
4. Reject solutions that add pause-era flow, unreadable action clutter, generic survivor behavior, or detached spreadsheet progression.
5. Define acceptance criteria in terms of timing, readability, input response, support clarity, and payoff.
6. Hand off to ALFRED for implementation or CYBORG for hardening.

## Output
- Current repo truth.
- Actual gap.
- Proposed bounded move.
- Non-goals.
- Validation plan.
