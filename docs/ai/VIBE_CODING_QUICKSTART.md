# WHAT WE FED — VIBE CODING QUICKSTART

Purpose: keep sessions high-power with low token load.

## Startup Sequence (Default)
1. Read `docs/ai/PROJECT_KERNEL.md`.
2. Identify task lane: combat, validation, canon, workflow, data/content.
3. Open only the mapped files from `docs/ai/CONTEXT_EXPANSION_MAP.md`.
4. Choose one bounded pass (avoid broad rewrites).
5. Implement with explicit non-goals.
6. Validate and report verified vs unverified.

## Default-Load Budget Rule
- Start with kernel + one route file by task lane.
- Expand only when unresolved conflict remains after the first pass.

## Bounded-Pass Rule
- Prefer one concrete move over multi-system churn.
- If task is medium/high mutation: compare safe path vs mutation path, then pick one recommendation.
- If task is low mutation (bug/QC/docs hygiene): stay single-path and non-speculative.

## Identity Preservation Rule
- Never trade timing truth, lane clarity, support readability, DNA meaning, or Bond/Eat tension for convenience.
- **Combat-clean, management-rich**: No menu sludge (repeated interruption) during active in-level combat; rich management is encouraged outside combat.
- **Display Law**: **Combat HUD = Urgency** (live action) | **Management Screens = Comprehension** (between-level strategy).
- Do not flatten tone into generic genre templates.

## Expansion Discipline
- Do not load full canon by default.
- Open flagship/deferred docs only when the task explicitly depends on later scope.
- Keep long doctrine files as deep reference, not startup baggage.

## Validation Discipline
- Runtime claim -> run repo validation commands where practical.
- If not runtime-tested, say so.
- Always list what remains unverified.

## Done Condition
- Change solves requested lane.
- Locked core preserved.
- Scope did not silently widen.
- Report includes: files touched, validation, risks, next bounded follow-up.
